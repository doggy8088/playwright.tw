(實驗性) Service Worker 網路事件

簡介​

warning

如果你正在尋找一般的網路模擬、路由和攔截，請先參閱 Network Guide。Playwright 提供內建的 API 用於此用途，不需要下面的資訊。然而，如果你對 Service Workers 自身發出的請求感興趣，請繼續閱讀。

Service Workers 提供了一種瀏覽器原生的方法來處理頁面使用原生 Fetch API (fetch) 所發出的請求以及其他網路請求的資產（如腳本、css 和圖片）。

他們可以作為頁面和外部網路之間的網路代理來執行快取邏輯，或者如果 Service Worker 添加 FetchEvent 監聽器，則可以為用戶提供離線體驗。

許多使用 Service Workers 的網站僅將其用作透明的最佳化技術。雖然用戶可能會注意到更快的體驗，但應用程式的實現並不知道它們的存在。啟用或未啟用 Service Workers 執行應用程式在功能上看起來是等效的。

如何啟用​

Playwright 的 Service Workers 所發出的請求檢查和路由是實驗性的，預設情況下是停用的。

將 PW_EXPERIMENTAL_SERVICE_WORKER_NETWORK_EVENTS 環境變數設置為 1（或任何其他值）以啟用此功能。目前僅支援 Chrome/Chromium。

如果你正在使用（或有興趣使用此功能），請在此問題上留言讓我們知道你的使用案例。

服務工作者抓取​

存取 Service Workers 和等待啟動​

你可以使用 browserContext.serviceWorkers() 來列出 Service Worker，或者如果你預期頁面會觸發其註冊，可以特別監看 Service Worker：

const serviceWorkerPromise = context.waitForEvent('serviceworker');
await page.goto('/example-with-a-service-worker.html');
const serviceworker = await serviceWorkerPromise;

browserContext.on('serviceworker') 是在 Service Worker 的主腳本被評估之前觸發的，所以在呼叫 serviceworker.evaluate() 之前，你應該等待它的啟動。

等待 Service Worker 被啟動有更多慣用的方法，但以下是一種與實作無關的方法:

await page.evaluate(async () => {
  const registration = await window.navigator.serviceWorker.getRegistration();
  if (registration.active?.state === 'activated')
    return;
  await new Promise(res =>
    window.navigator.serviceWorker.addEventListener('controllerchange', res),
  );
});

網路事件和路由​

任何網路請求由 Service Worker 發出將會有:

  • browserContext.on('request') 和其對應的事件(browserContext.on('requestfinished') 和 browserContext.on('response'), 或 browserContext.on('requestfailed'))
  • browserContext.route() 會看到該請求
  • request.serviceWorker() 將被設置為 Service Worker 實例, 並且 request.frame() 將拋出
  • response.fromServiceWorker() 將返回 false

此外，任何由 Page（包括其子 Frame）發出的網路請求將會有:

  • browserContext.on('request') 和其對應的事件(browserContext.on('requestfinished') 和 browserContext.on('response'), 或 browserContext.on('requestfailed'))
  • page.on('request') 和其對應的事件(page.on('requestfinished') 和 page.on('response'), 或 page.on('requestfailed'))
  • page.route() 和 page.route() 將不會看到請求(如果已註冊了 Service Worker 的 fetch 處理器)
  • request.serviceWorker() 將被設置為 null, 而 request.frame() 將返回 Frame
  • response.fromServiceWorker() 將返回 true(如果已註冊了 Service Worker 的 fetch 處理器)

許多 Service Worker 實作只是簡單地執行來自頁面的請求（可能為了簡單起見省略了一些自訂的快取/離線邏輯）:

transparent-service-worker.js
self.addEventListener('fetch', event => {
  // actually make the request
  const responsePromise = fetch(event.request);
  // send it back to the page
  event.respondWith(responsePromise);
});

self.addEventListener('activate', event => {
  event.waitUntil(clients.claim());
});

如果一個頁面註冊了上述的 Service Worker:

<!-- filename: index.html -->
<script>
  window.registrationPromise = navigator.serviceWorker.register('/transparent-service-worker.js');
</script>

在第一次通過 page.goto() 訪問頁面時，將會發出以下請求/回應事件（以及相應的網路生命週期事件）：

事件擁有者URL路由response.fromServiceWorker
browserContext.on('request')Frameindex.html是
page.on('request')Frameindex.html是
browserContext.on('request')Service Workertransparent-service-worker.js是
browserContext.on('request')Service Workerdata.json是
browserContext.on('request')Framedata.json是
page.on('request')Framedata.json是

由於範例 Service Worker 只是作為基本透明的 "代理":

  • 有兩個 browserContext.on('request') 事件針對 data.json; 一個是 Frame 擁有的，另一個是 Service Worker 擁有的。
  • 只有 Service Worker 擁有的資源請求可以通過 browserContext.route() 進行路由; 針對 data.json 的 Frame 擁有事件是無法路由的，因為它們甚至沒有可能訪問外部網絡，因為 Service Worker 已經註冊了一個 fetch 處理器。
caution

重要的是要注意: 呼叫 request.frame() 或 response.frame() 將會 拋出 一個例外，如果在一個具有非空 request.serviceWorker() 的 Request/Response 上呼叫。

高級範例​

當 Service Worker 處理頁面的請求時，Service Worker 可以向外部網路發出 0 到 n 個請求。Service Worker 可能直接從快取回應、在記憶體中生成回應、重寫請求、發出兩個請求然後合併成一個等。

考慮以下的程式碼片段來了解 Playwright 對 Request/Responses 的看法，以及它如何影響這些情況下的路由。

complex-service-worker.js
self.addEventListener('install', function(event) {
  event.waitUntil(
      caches.open('v1').then(function(cache) {
      // 1. Pre-fetches and caches /addressbook.json
        return cache.add('/addressbook.json');
      })
  );
});

// Opt to handle FetchEvent's from the page
self.addEventListener('fetch', event => {
  event.respondWith(
      (async () => {
        // 1. Try to first serve directly from caches
        const response = await caches.match(event.request);
        if (response)
          return response;

        // 2. Re-write request for /foo to /bar
        if (event.request.url.endsWith('foo'))
          return fetch('./bar');

        // 3. Prevent tracker.js from being retrieved, and returns a placeholder response
        if (event.request.url.endsWith('tracker.js')) {
          return new Response('console.log("no trackers!")', {
            status: 200,
            headers: { 'Content-Type': 'text/javascript' },
          });
        }

        // 4. Otherwise, fallthrough, perform the fetch and respond
        return fetch(event.request);
      })()
  );
});

self.addEventListener('activate', event => {
  event.waitUntil(clients.claim());
});

而且一個簡單註冊 Service Worker 的頁面:

<!-- filename: index.html -->
<script>
  window.registrationPromise = navigator.serviceWorker.register('/complex-service-worker.js');
</script>

在第一次通過 page.goto() 訪問頁面時，將會發出以下的請求/回應事件：

事件擁有者URL路由response.fromServiceWorker
browserContext.on('request')Frameindex.html是
page.on('request')Frameindex.html是
browserContext.on('request')Service Workercomplex-service-worker.js是

需要注意的是 cache.add 使得 Service Worker 發出請求(Service Worker-owned)，即使在頁面中請求 addressbook.json 之前。

一旦 Service Worker 被啟用並處理 FetchEvents，如果頁面發出以下請求:

await page.evaluate(() => fetch('/addressbook.json'));
await page.evaluate(() => fetch('/foo'));
await page.evaluate(() => fetch('/tracker.js'));
await page.evaluate(() => fetch('/fallthrough.txt'));

以下的請求/回應事件將會被觸發:

browserContext.on('request')Frameaddressbook.jsonYes
page.on('request')Frameaddressbook.jsonYes
browserContext.on('request')Service WorkerbarYes
browserContext.on('request')FramefooYes
page.on('request')FramefooYes

請注意:

  • 該頁面請求了 /foo, 但服務工作者請求了 /bar, 因此只有 Frame-擁有的事件是針對 /foo, 而不是 /bar。
  • 同樣地, 服務工作者從未訪問過 tracker.js 的網絡, 因此只有 Frame-擁有的事件是針對該請求發出的。

僅路由服務工作者請求​

await context.route('**', async route => {
  if (route.request().serviceWorker()) {
    // NB: calling route.request().frame() here would THROW
    return route.fulfill({
      contentType: 'text/plain',
      status: 200,
      body: 'from sw',
    });
  } else {
    return route.continue();
  }
});

已知限制​

目前無法路由更新的 Service Worker 主腳本程式碼的請求 (https://github.com/microsoft/playwright/issues/14711)。
